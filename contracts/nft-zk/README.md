# NFT-ZK Contract

A privacy-preserving NFT implementation for the Midnight blockchain, written in the Compact language.
Ownership is stored as a per-token cryptographic **commitment** rather than a public address, so the
public ledger never records who owns what, and two tokens held by the same person cannot be linked.

> For a simpler transparent alternative, see [NFT](../nft/).

## Structure

```
contracts/nft-zk/
├── src/
│   ├── nft-zk.compact             # Example contract (secret-derived admin: mintAdmin / rotateAdmin)
│   ├── witnesses.ts               # TypeScript witness definitions
│   ├── modules/
│   │   └── NftZk.compact          # Reusable NFT-ZK module (commitments, release/claim, metadata)
│   └── test/
│       ├── nft-zk-simulator.ts    # Contract simulator
│       └── nft-zk.test.ts         # Test suite (multi-party + adversarial cases)
├── CONTRACT.md
├── KNOWN_LIMITATIONS.md
└── README.md
```

## The modular pattern

The core NFT-ZK logic lives in `modules/NftZk.compact` and exports its circuits with no authorization.
The example contract in `nft-zk.compact` imports the module and adds a **secret-derived admin** as one
way to gate issuance. You decide how to authorize the circuits in your own contract.

Note the module also defines a raw `mint` circuit, but the example contract **deliberately does not export
it** (only the admin-gated `mintAdmin` is callable). Do not add `mint` to the wrapper export list.

## Privacy model

Token ownership is stored as a per-token **owner commitment**, not an address:

```
persistentHash(["nftzk:owner:v1", pubkey, local_secret, tokenId])
```

Folding `tokenId` into the hash means each of a holder's tokens produces a **different** on-chain
commitment, so an observer cannot group a person's holdings. Only the holder can reproduce the commitment,
because it depends on a `local_secret` that never leaves their device (a witness).

- **No on-chain `balanceOf`.** A public per-user count would itself leak ownership, so balance and
  enumeration are an offline/wallet concern (recompute your commitments locally and check the indexer).
- **Metadata** is a public per-token URI (`tokenUri`), set at mint and carried through transfers.
- **Two secrets** back the flows: `getLocalSecret()` (the owner's self secret) and `getClaimSalt()` (a
  fresh per-transfer salt generated by the recipient).

## Transfers (two step: release then claim)

Transfers never record an "A sent to B" link. The recipient first generates a **claim commitment**
`persistentHash(["nftzk:claim:v1", recipient_pubkey, claim_salt])` off-chain and shares it with the
sender. Then:

1. `release(tokenId, claim_commitment)` — the owner proves ownership and moves the token into a pending
   claim under the recipient's commitment.
2. `claim(tokenId)` — the recipient proves they are the party behind the claim commitment and takes
   ownership under their own per-token owner commitment.

## Circuits

| Circuit | Description |
|---|---|
| `commitOwner(pk, secret, tokenId)` | Pure. Build a per-token owner commitment. |
| `commitClaim(pk, salt)` | Pure. Build a claim (in-transit) commitment. |
| `tokenExists(tokenId)` | Whether a token is owned or pending. |
| `ownerOf(tokenId)` | The owner commitment of an owned token (a hash, reveals no identity). |
| `pendingOf(tokenId)` | The claim commitment of a pending token. |
| `uriOf(tokenId)` | The token's metadata URI. |
| `release(tokenId, claim_commitment)` | Owner releases a token into a recipient's claim commitment. |
| `claim(tokenId)` | Recipient claims a pending token under their own owner commitment. |
| `burn(tokenId)` | Owner burns their own token and its metadata. |
| `mint(claim_commitment, tokenId, uri)` | Create a new token *(module only, not exported by the example, wrap with your auth)*. |

The example contract (`nft-zk.compact`) adds:

| Circuit | Description |
|---|---|
| `mintAdmin(claim_commitment, tokenId, uri)` | **[admin only]** Mint into a recipient's claim commitment. |
| `rotateAdmin(newAdmin)` | **[admin only]** Hand issuance authority to a new admin public key. |
| `deriveAdminPublicKey(sk)` | Pure. Derive the admin public key from the admin secret. |

## Authorization note

Admin authorization is **secret-derived**: the contract stores only the derived admin public key on the
ledger, and each admin circuit proves knowledge of the admin secret (`contractAdmin ==
deriveAdminPublicKey(localSecretKey())`). It does **not** authorize with `ownPublicKey()`, which is a
prover-supplied witness and not a trustworthy caller identity. See [CONTRACT.md](./CONTRACT.md) and
[KNOWN_LIMITATIONS.md](./KNOWN_LIMITATIONS.md).
